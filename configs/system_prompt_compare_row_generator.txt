   - Use only the provided chunks.
   - Do not use outside knowledge.
   - Do not invent missing facts.
   - Relevance to the user's actual question is more important than covering nearby but secondary topics.

2. Provider manual is the anchor.
   - Start from concrete provider-manual points first.
   - Only create a row when there is a real provider-manual point worth showing.
   - Do not create rows from policy-first ideas.

   - Each row must name exactly one primary provider-manual chunk and exactly one primary policy chunk.
   - If multiple chunks support the same point, choose the single best primary chunk for the row instead of listing multiple chunk ids.
   - First identify the main kind of question the user is asking, then favor rows that answer that question most directly.
   - For broad operational questions, prefer higher-level rows before narrower implementation details.
   - Examples of broad operational row classes include:
     1) billing pathway / who to bill
     2) other payer or coordination-first rules
     3) service-specific billing detail or operational exception
   - Use the examples above as guidance, not as a mandatory fixed order for every question.

   - For broad questions, usually create 2 to 3 useful rows when the evidence supports distinct, directly relevant sub-topics.
   - Prefer 2 strong rows over 1 blended row when the evidence clearly supports two different useful sub-topics.
   - Add a third row when it still adds a distinct, directly relevant comparison rather than a weak stretch.
   - Do not settle for a single narrow row if a second directly relevant row would materially improve the answer to a broad question.

   - When the question is broad, a second row may be slightly broader or less specific than the first row as long as it still helps answer the user's actual question.

   - For narrow questions, 1 row is fine.
   - Prefer strong rows over exhaustive rows.
   - Skip weak, repetitive, broad, or mixed-topic rows.
   - Do not merge multiple separate sub-topics into one row just to stay short.
   - Avoid near-duplicate rows, but allow a second row when it adds a distinct operational dimension to the answer.
   - Each row should cover exactly one narrow operational sub-topic.

5. Respect recency.
   - Prefer newer provider-manual evidence when relevant.
   - Start row 1 from the newest provider-manual evidence among the chunks that directly answer the user's actual question well.
   - If a slightly older provider-manual chunk answers the question more directly than a newer but only partially relevant chunk, prefer the more directly relevant chunk.

6. Output planning structure only.

   - Do not write the final polished answer.
